Prompt engineering assumes a human writes the next instruction. Loop engineering assumes nobody does — the agent has to generate its own next instruction, on a schedule, in response to an event, or until some condition is met, and keep doing that without a person in the middle of every cycle.

## The shape underneath every version of this

Strip away the framework-specific naming and almost every production agent loop reduces to the same four moves: discover what state the world is actually in, plan the next action against that state and the goal, execute it, verify whether it actually worked — then decide whether to continue, stop, or escalate to a human.

The research literature has converged on this same shape from a few different directions, and the differences between the versions are the actual design decisions. [ReAct](https://arxiv.org/abs/2210.03629) interleaves reasoning and acting turn by turn — the model reasons, acts, observes what came back, reasons again — and that tight interleaving is what let it cut hallucination and error propagation on HotpotQA and Fever versus plain chain-of-thought, and beat imitation and RL baselines by 34 and 10 absolute points of success rate on ALFWorld and WebShop using only one or two in-context examples. That's a real result, and it's also a specific bet: the plan gets rewritten every step, based on whatever the last action returned.

[Plan-then-Execute](https://arxiv.org/abs/2509.08646) makes the opposite bet on purpose. A planner produces the full plan up front; an executor carries it out without renegotiating the plan mid-flight. That trade buys predictability and modularity, and it buys something a reactive, ReAct-style loop structurally can't: control-flow integrity. If a tool call returns adversarial or just malformed output, a step-by-step reasoning loop can be talked into a new plan by that output. A locked plan can't — there's nothing for an injected instruction to attach to. That's why the pattern shows up in security-focused agent architectures, not just reliability-focused ones. Extend it once more to [Plan-Validate-Execute](https://community.sap.com/t5/security-and-compliance-blog-posts/plan-then-execute-an-architectural-pattern-for-responsible-agentic-ai/ba-p/14239753) and you get a Verifier stage that checks the plan itself — logical coherence, policy compliance, system constraints — before a single action runs, which catches planner overconfidence before it becomes an executed mistake rather than after.

None of these three architectures is "correct" independent of the task. ReAct is right when the environment is genuinely unpredictable and the value is in adapting fast. Plan-then-Execute is right when you need to guarantee the agent can't be steered off its objective mid-task, which is most of what matters once a loop is touching production data or taking external actions. Anthropic's own [agent-design guidance](https://www.anthropic.com/research/building-effective-agents) draws this same line near the top: a workflow is predefined code paths, an agent is an LLM dynamically directing its own process and tool use, and the more autonomy you hand the loop, the more explicit its stopping conditions need to be — a maximum iteration count, a checkpoint where it pauses for a human, sandboxed testing — because autonomy compounds errors exactly as fast as it compounds progress.

## A stop condition is a thing a machine can check, not a thing a person would say

"Keep going until it's good" is not a stop condition. It's a sentence that sounds like one. The distinction that actually matters is whether the check is deterministic — true or false, evaluated the same way on iteration one and iteration two hundred — or whether it depends on a judgment call the loop re-makes every cycle. [MindStudio's design guidance](https://www.mindstudio.ai/blog/agent-loops-verifiable-stop-conditions) documents what happens with the second kind: agents exit too early, too late, or run indefinitely, because "good enough" gets re-interpreted every pass instead of checked against a fixed bar. The patterns that hold up are boring on purpose — count-based caps, a state-change condition grounded in something external to the model's own opinion, a list or batch that's either fully processed or it isn't, a numeric threshold fixed before the loop starts, schema validation. All of them share one property: the loop doesn't get to decide, mid-run, what counts as done.

The failure mode on the other side of this is documented, not hypothetical. [MindStudio records a case](https://www.mindstudio.ai/blog/agent-loops-explained-trigger-action-stop-condition) of an agent told to "retry until data was retrieved," with no hard stop condition, that called a broken tool 400 times in five minutes. A max-iteration limit of three would have caught it before the first minute was up. The rule that falls out of that: every loop needs a verifiable stop condition as the primary exit, and a maximum iteration count as the fallback for when the primary condition is wrong, gets gamed, or simply never fires the way you expected.

## The cap and the stop condition are two different levers, not one

It's tempting to treat the iteration cap as a backstop for the stop condition and leave it there — set the cap generously, trust the stop condition to fire first. That conflates two different failure modes that need two different limits. Anthropic's [task_budget](https://platform.claude.com/docs/en/build-with-claude/task-budgets) feature makes the distinction explicit in the API itself: it gives the model an advisory token budget spanning the whole loop — thinking, tool calls, tool results, output — with a countdown marker visible only to the model, so it can pace itself and wrap up gracefully as the budget depletes instead of getting cut off mid-action. Crucially, that's a soft hint. The hard ceiling is a separate parameter, max_tokens, on the individual request, and the two are deliberately independent, because they answer different questions: the budget answers "how should the model pace this specific run," the hard cap answers "what's the absolute worst case if pacing fails." Collapsing them into one number means you can't tune either without breaking the other.

The failure mode when the sizing is wrong is instructive too. Anthropic's own documentation notes that a task_budget set too small for the actual work — 20,000 tokens for a job that needs hours — doesn't just get truncated cleanly. The model may decline the task outright, scope it down aggressively, or stop early with a partial result, none of which look like the deliberate stop you designed for. A mis-sized cap doesn't fail safe by default; it fails in whatever shape falls out of the model's own judgment about how to handle running short — exactly the kind of unverifiable judgment call a real stop condition is supposed to remove from the loop in the first place.

## The verifier can't be the generator wearing a different hat

This is the part of loop design most teams under-invest in, because it looks like it should be free — the same model that just produced the output can presumably also check it. Sometimes that's true. [Self-Refine](https://arxiv.org/abs/2303.17651) has a model generate, critique its own output, and revise, with no external feedback and no additional training, and gets roughly 20% absolute average improvement across seven tasks, with outputs humans and automatic metrics both preferred over the one-shot version. That's a real result, and it's the reason "just ask it to check its own work" isn't a crazy default.

But it has a boundary, and the boundary is exactly where loop design earns its keep. [Huang et al.](https://arxiv.org/abs/2310.01798) found that intrinsic self-correction — a model fixing its own reasoning using nothing but its own internal capability, no external signal — is fundamentally limited: models struggle to self-correct without outside feedback, and accuracy can drop after a self-correction pass rather than improve. A [related survey](https://direct.mit.edu/tacl/article/doi/10.1162/tacl_a_00713/125177/When-Can-LLMs-Actually-Correct-Their-Own-Mistakes) names the specific shape of the blind spot: a model can correct an error when it's pointed out externally — by a tool, by a different model — and fail to correct the identical error sitting in its own prior output. Same mistake, different fate, depending entirely on who's holding up the mirror. Majority-vote self-consistency doesn't route around this either: [a newer analysis](https://arxiv.org/pdf/2511.00751) found intra-model majority voting produced under 1% improvement on current frontier models, and, more damning, that a model can be near-perfectly self-consistent while being consistently wrong. Agreement across samples from the same weights isn't evidence of correctness — it's evidence the same blind spot fired every time.

I've watched this exact failure shape at a smaller scale, building Zen AI, the low-code agent platform that took us to 300+ agents in nine months. A loop that graded its own completion — "did I finish the thing I was asked to do," answered by the same model that did the thing — drifted the way the research predicts: confidently wrong, not obviously wrong. What fixed it wasn't a better prompt asking the model to be more critical of itself. It was moving the check outside the model entirely: a policy rule, a schema check, a deterministic test that either passes or doesn't. That's the same principle underneath TDA's compliance gates today — TDA being the Tata Steel Digital Assistant, the enterprise RAG assistant now rolled out to 76,000+ people, 13,000+ daily active, across 5 Tata Group companies. Those gates don't ask the model whether an action is compliant; they check it against a rule enforced in code, which doesn't carry the model's blind spots because it isn't a model.

## Escalation is a security property, not a UX nicety

The last non-negotiable is what happens when the loop genuinely can't meet the goal, and the reason this deserves real design attention rather than a generic "ask for help" fallback is that a loop's own sense of whether it's done is an attack surface, not just an occasionally-wrong opinion. [LoopTrap](https://arxiv.org/html/2605.05846v1) is a documented attack that targets exactly this: when an agent self-evaluates task completion, an adversary can inject content that distorts that judgment, convincing the agent a finished task is incomplete and driving unbounded extra computation — demonstrated across 8 mainstream agents with an average 3.57x amplification in steps taken. The defense isn't a smarter self-assessment prompt. It's the same move as the verifier problem above: termination judgment needs an external, independent check, because a loop that trusts its own opinion about whether it's finished can be made to hold the wrong opinion by something outside its control.

The same logic applies to how escalation gets designed once a loop does correctly decide it's stuck. [Escalation design guidance](https://www.digitalapplied.com/blog/human-in-the-loop-escalation-design-ai-agents-2026) argues for tiering actions by risk and reversibility — read-only work fully autonomous, reversible actions autonomous with logging, external actions gated to a staging queue, anything high-risk or irreversible requiring mandatory human approval fixed into the workflow definition, not something the agent can negotiate at runtime. The reason it has to be fixed rather than confidence-gated is uncomfortable but well documented: models are systematically miscalibrated — a claimed 90% confidence can reflect closer to 75% real accuracy — and that miscalibration compounds across steps. A three-step chain, each step honestly claiming 90% confidence, nets out to roughly 42% actual end-to-end reliability. Gating escalation on the agent's stated confidence just adds another self-reported number to a chain that's already unreliable. A good handoff instead bundles a plain-language description of the action, the reasoning behind it, an impact and reversibility estimate, alternatives considered, and a session ID for audit correlation — enough for a human to make the call without reconstructing the whole run first.

## The loop actually running behind this post

The AI-focused posts on this site are drafted by exactly this kind of loop, and it's a genuinely low-stakes version of everything above, which is exactly why it's a clean place to see the pattern. A scheduled job pulls the next topic from a queue, drafts a post, and opens it as a pull request. It never publishes directly. The stop condition is verifiable in the boring sense the research above argues for — "a draft PR exists" — not "the post is good," which is a judgment call I'm not willing to hand to the loop about writing that goes out under my name. That judgment is the human checkpoint, on purpose, and it's a mandatory gate rather than a confidence-gated one, for the same reason risk-tiered escalation design argues for mandatory gates on anything that isn't cheaply reversible: a bad draft costs a review cycle; a bad post published under my byline isn't something a retry fixes.

It's a much smaller version of the same discipline that had to exist before Zen AI could run 300+ agents safely across nine months, and before TDA could reach 76,000+ people and 13,000+ daily active users across five Tata Group companies at 99% uptime — cleared through VAPT (vulnerability assessment and penetration testing) security review, running on RBAC (role-based access control) and encryption enforced in code, not requested in a prompt. The scale is different. The design question underneath it isn't: what can this loop check for itself, but what has to be checked by something, or someone, that doesn't share its blind spot.
