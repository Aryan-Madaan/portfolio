Swap the model behind most production agent failures and the failure usually persists. That's the tell. If a different model, from a different lab, with a different training run, reproduces the same broken behavior against the same task, the model was never the variable that mattered. Something around it was.

A model is a function that maps context to a next action. A harness is everything that decides what context it sees, what it's allowed to do with that action, and what happens when it gets it wrong. Two teams can point the identical model at the identical problem and get completely different reliability, because the harness — not the model — is doing most of the work.

This isn't a hunch. On [APEX-Agents](https://medium.com/@epappas/the-agent-harness-is-the-architecture-and-your-model-is-not-the-bottleneck-5ae5fd067bb2), a 480-task benchmark built from real professional work, the best agent managed only 24% pass@1 (about 40% at pass@8). The dominant failure category wasn't missing knowledge — it was execution and orchestration: agents that had the right facts but got lost after too many steps and looped back into approaches that had already failed. [Vercel ran a cleaner, if much smaller, experiment on the same question](https://vercel.com/blog/we-removed-80-percent-of-our-agents-tools): on one internal text-to-SQL agent, tested against a five-query eval, they cut the tool surface from 15 specialized tools down to 2 general-purpose ones — bash and file access — with the exact same underlying model, and accuracy went from 4/5 to 5/5, token use dropped 37%, and it ran 3.5x faster. It's an n of 5, not a benchmark, so take it as a clean illustration rather than proof — but the direction matches APEX-Agents at 480 tasks: nothing about the model changed, only the harness did. A [recent paper](https://arxiv.org/pdf/2605.23950) makes the uncomfortable version of this point explicit: harness engineering — prompting strategy, tool implementation, error recovery — contributes more variance to benchmark scores than the choice of underlying model, and almost no published agent benchmark discloses harness details. Most agent leaderboards are quietly comparing harnesses, not models.

## Five layers, and where each one actually breaks

A harness is five layers: tool orchestration, verification loops, context and memory, guardrails, observability. I didn't arrive at this list by brainstorming a framework — I arrived at it by watching which layer was actually responsible each time something broke at Tata Steel, first building Zen AI — a low-code platform that took us to 300+ agents in 9 months — and then TDA, an enterprise assistant now rolled out to 76,000+ people, with 13,000+ daily active users across 5 group companies. Each layer below is the one that failed, and the specific mechanism by which it failed.

**Tool orchestration** decides what the agent can touch, in what order, and — critically — what happens when one tool call returns something wrong instead of failing loudly. The taxonomy in [Why Do Multi-Agent LLM Systems Fail?](https://arxiv.org/html/2503.13657v1) names the exact shape of this: a verifier that "only checks if code compiles without running the program" lets a broken output pass downstream as if it were correct, and an agent that proceeds on an input it should have flagged as uncertain (the paper calls this failing to ask for clarification) cascades that error through every step after it. A [survey of multi-tool orchestration](https://arxiv.org/pdf/2603.22862) documents this same general shape from the state-management side: once one tool call's parameters drift from what the task actually needs, every later step that trusts that state as ground truth inherits the error, with nothing built in to catch it. This is also why weaker harnesses get stuck in loops: [SciAgentGym](https://arxiv.org/pdf/2602.12984), a benchmark of 1,780 scientific tools across four disciplines, finds that agent performance on multi-step scientific tool-use collapses as the number of required steps grows — the same shape as an orchestration failure, not a knowledge one, since the facts the agent needs don't change, only the number of steps it has to chain together correctly does. When we scaled Zen AI past a hundred agents, this is precisely where hand-holding stopped working: you cannot manually inspect every tool call across 300 agents, so the orchestration layer has to fail loud and fail closed by construction, or you find out about the failure from a user instead of a log.

**Verification loops** are supposed to be the fix for exactly this — something outside the model confirming a step actually happened, rather than trusting the model's claim that it did. Self-feedback genuinely works under the right conditions: [Self-Refine](https://arxiv.org/abs/2303.17651) shows the same model can improve its own output by roughly 20% absolute across seven tasks using nothing but self-generated critique, no external supervision at all. But that result has a boundary, and the boundary is the whole design problem: when the generator and the verifier are the literal same weights, they carry the same blind spots into both roles, so a model can wave through in its own output exactly the kind of error it would catch if someone else had written it. [Snorkel's controlled test](https://snorkel.ai/blog/the-self-critique-paradox-why-ai-verification-fails-where-its-needed-most/) puts a number on how badly this can go: across a 50-task visual-reasoning eval with Claude Sonnet 4.5 and OpenAI o4-mini, self-critique dropped Sonnet's accuracy on tasks it already solved reliably from 98.1% to 56.9%, because a critic primed to find flaws hallucinated flaws that weren't there — while the same technique lifted its accuracy on tasks it was failing from 0% to 60%. Same technique, opposite effect, depending entirely on whether the critic is independent of the generator's blind spot. This is also why LLM-as-judge setups quietly fail when judge and generator share a model family: shared training distributions mean shared inductive biases, so the judge scores an error highly precisely because it can't recognize it as an error — the same convergence problem showing up again, just with the judge and generator as two separate calls instead of two roles played by one. Majority-vote self-consistency has the same failure shape: [Too Consistent to Detect](https://arxiv.org/pdf/2505.17656) shows models can confidently reproduce the same wrong answer across every sampled path, which defeats a variance-based check entirely — agreement isn't evidence of correctness when the error is shared across samples, not random. The design rule that falls out of all of this: a verification loop only earns its keep if it gets a genuinely independent signal — actually running the code instead of checking that it compiles, a different model, a rule, a human — not another look from the same weights that made the mistake.

**Context and memory** is where a harness accumulates and carries state across a long task, and it fails silently, which is worse than failing loudly. Compaction — summarizing older turns to fit a token budget — is lossy by design: a mis-summarized variable name, constraint, or API signature gets carried forward as if it were ground truth, and there's no later query that can recover the information that summarization destroyed. [One practitioner breakdown of the failure mode](https://blakecrosley.com/blog/agent-context-compaction) makes the same point from the timing angle: compact in the middle of a task, not at a clean boundary, and you don't just lose color — you can force the agent to re-derive work it had already finished, because the compacted note lands mid-derivation instead of after it. The sharpest version of this is [Governance Decay](https://arxiv.org/pdf/2606.22528): safety directives placed early in a system prompt get no special protection during standard compression, so they're discarded like anything else — the agent keeps behaving competently while its actual constraints have silently been erased over a long session, with no error, no warning, nothing to signal that it happened. The fix isn't "compact less" — it's treating memory as a designed lifecycle. [Production guidance from Redis](https://redis.io/blog/context-compaction/) draws the distinction that matters: reversible compaction, where the dropped content still exists elsewhere and can be pulled back on demand, versus lossy summarization that destroys it outright — with the compaction policy itself built to explicitly preserve active constraints the agent is still bound by, open decisions not yet acted on, and completed-task state, and only discard the exploratory process and redundant tool output that led there. At the scale TDA runs at, you cannot rely on an agent's implicit sense of what matters in its own history — the lifecycle has to be explicit, or the erosion is invisible until an incident.

**Guardrails** are the layer that has to hold regardless of what the model decides, and the underlying reason is architectural, not a matter of prompt discipline. LLMs have no built-in mechanism to separate privileged system instructions from untrusted user or tool content — it's all one token stream, and delimiters or role markers are a convention the model usually follows, not a boundary it's structurally forced to respect. That gap is exactly why prompt injection sits at #1 in OWASP's Top 10 for LLM applications, and why published attack success rates against agents with auto-execution capability run high enough — commonly well over 50%, and higher still against undefended systems — that no single prompt-level mitigation should ever be trusted as the last line of defense. The [guardrail literature](https://budecosystem.com/a-survey-on-llm-guardrails-methods-best-practices-and-optimisations/) is blunt about the implication: packing guardrail policy into the prompt alone doesn't just under-protect, it's expensive — one analysis cited there found it can triple latency and cost — and there's still no formal guarantee that an injected instruction loses to an adversarial one. So the only real answer is layered defense: prompt guidance plus classifiers plus rules enforced in code that no amount of clever prompting can talk its way past. This is exactly why TDA's access model is RBAC, encryption, and a VAPT security clearance sitting in code and infrastructure, not in a system prompt telling the model to behave — it's also why that's the layer that let it clear security review for 76,000 people across five group companies in the first place.

**Observability** is the layer that turns "something went wrong somewhere in there" into an actual answer. [OpenTelemetry's GenAI semantic conventions](https://opentelemetry.io/blog/2026/genai-observability/), backed by CNCF, standardize an `invoke_agent` span containing child spans for each model call and each `execute_tool` call, with attributes like `gen_ai.request.model`, input and output token counts, finish reasons, and — when enabled — the actual input and output messages. That turns "was it the model, the tool call, or a retry loop" from a guess into an inspectable trace. This isn't standing still: [OpenTelemetry's own agent-observability initiative named CrewAI, AutoGen, LangGraph, IBM Bee Stack, and IBM wxFlow](https://opentelemetry.io/blog/2025/ai-agent-observability/) as the frameworks it wanted to standardize this convention for, and a year on, [Datadog now has confirmed native support for v1.37+ of the spec](https://greptime.com/blogs/2026-05-09-opentelemetry-genai-semantic-conventions) — the OpenAI Python SDK's instrumentation is the most mature piece, while LangGraph's and CrewAI's are still explicitly in progress, not finished. The GenAI conventions themselves remain in Development status with no public stabilization date. Progress is real; "everyone's already converged on this" still isn't true. Directionally, though, the industry has independently concluded that agent failures are mostly pipeline failures, and pipeline failures need pipeline-level visibility, not single-call visibility. Go back to APEX-Agents: the failure mode was agents getting lost after too many steps and looping back to dead ends. You cannot fix "got lost" without a trace that shows you exactly where it got lost. TDA's 99% uptime isn't a model property — it's what you get when you can see every layer above this one clearly enough to fix the right thing before a user notices.

## The one habit that generalizes

Every layer above has the same shape: a specific, well-documented failure mode, and a specific mechanism that closes it. The discipline that actually matters is refusing to treat any of these as a prompting problem. When an agent fails in a way you didn't anticipate, the fix is never "tell it to be more careful" — it's identifying which of the five layers let that failure through, and changing that layer's mechanism so the same class of mistake becomes structurally harder to make again. A verifier that only checked compilation gets rewritten to actually run the test. A tool that failed silently gets rewritten to fail loud. A compaction policy that dropped a constraint gets rewritten to protect it explicitly. That's also where you decide, in the guardrail layer specifically and not as an afterthought, which categories of action never get delegated at all — an approval gate before anything irreversible, a hard rule enforced in code rather than requested in a prompt.

None of the five layers above is optional past a certain scale, and none of them is where most teams start. They're also, not coincidentally, most of what it actually took to get Zen AI to 300+ agents in 9 months and TDA to 76,000+ rolled out with 13,000+ people using it daily across 5 group companies, at 99% uptime, cleared through VAPT. The model choice was the easy 10% of that. The rest was this.
